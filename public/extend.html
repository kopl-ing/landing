Extending Kopling

Section 1

What an extension is

An extension is PHP and templates. Full stop.

No build step, no JavaScript framework to learn, no upgrade roulette. If you've ever built a small Laravel package, you already know almost everything you need — Kopling's extension system leans on ordinary Composer and Laravel conventions wherever it can, and only introduces something new where the alternative would be worse.

Concretely: an extension is a normal Composer package. It gets discovered, its directories get auto-wired into Laravel, and it can declare a small number of specific capabilities — that's the whole shape of the system this page documents.

Section 2

Getting discovered

Two things in your extension's composer.json are what make it a Kopling extension at all:

type

"type": "kopling-extension" — this is how Kopling finds your package among everything else installed via Composer. Nothing else looks for a Kopling extension any other way.

autoload.psr-4

A normal PSR-4 mapping, e.g. "YourVendor\\YourExtension\\": "src/" — the same block every Composer package already needs for its classes to load.

That's the entire registration step. There's no separate file to list your extension in, no id to remember and repeat elsewhere — your entry-point class is derived automatically from that same PSR-4 namespace (see the next section), and every directory convention is found by presence, not by declaration.

Section 3

The Extension class

One file is required: src/Extension.php, always named Extension, always in the namespace your autoload.psr-4 block declares.

// src/Extension.php

namespace YourVendor\YourExtension;

use Kopling\Core\Extension\AbstractExtension;

class Extension extends AbstractExtension
{
    public static function name(): string
    {
        return 'Your Extension';
    }

    public static function description(): string
    {
        return 'One sentence describing what it does.';
    }
}

Two things worth understanding about the shape of this class, not just its syntax:

It's not a Laravel ServiceProvider

AbstractExtension has no relationship to Illuminate\Support\ServiceProvider at all. Laravel is the mainstream tool Kopling is built on, but extensions code against Kopling's own contract, never against Laravel directly — that way, a Laravel major version upgrade is a Kopling concern to absorb, never something every installed extension has to deal with itself. The actual Laravel wiring (registering views, routes, migrations, translations) happens inside Kopling core, in code you never write.

Why the base class isn't named Extension too

Every extension names its own class Extension — deliberately, so you never have to invent or remember a class name. That only works because the class it extends is named AbstractExtension, not Extension: if the base class shared that name, your own file would need an import alias to tell the two apart. One small naming choice removes an entire category of boilerplate you'd otherwise hit on every extension you ever write.

Kopling's own classes are never marked final. You can always extend or override, on purpose — that stays true everywhere in Kopling, not just here.

Section 4

Directory conventions

Everything below works by a directory simply existing in your extension, next to src/. Nothing to declare, no interface to implement, no method to call — put files where they belong and Kopling registers them for you.

PathWhat it doesNotes
views/Blade templatesAuto-registered under a namespace built from your full package name, vendor included, hyphenated — yourvendor/example becomes yourvendor-example::, so a template renders as view('yourvendor-example::hello'). The vendor stays in the namespace on purpose: two different vendors can each publish an extension called example, and dropping the vendor would make their view/translation namespaces collide.
lang/TranslationsSame auto-registered namespace as views — __('yourvendor-example::messages.hello').
migrations/Database migrationsAuto-registered, no database/ wrapper directory needed — just migrations/ directly.
routes/web.phpRoutesAuto-required — but not auto-wrapped in the web middleware group. Your file needs its own Route::middleware('web')->group(...), the same way Kopling core's own routes file does.
css/Plain, hand-written CSSNo Tailwind, no build step. For anything beyond a safelisted layout-utility subset (flex/grid/gap/spacing), reach for a <x-k::*> component instead of a custom style.
js/Plain ES modulesNo bundler required — a plain <script type="module"> file is enough for the rare case that genuinely needs one.

Both css/ and js/ are detected today; linking them onto the actual page is still an open item — see Section 8.

Section 5

Contracts

Directory conventions cover everything Kopling can infer from your extension's filesystem. A small number of capabilities can't be expressed that way — declaring that your extension provides a storage driver, for instance, isn't something a folder existing can tell us. For those, implement one of Kopling's contract interfaces on your Extension class:

use Kopling\Core\Extension\Contract\RequestsStorageDriver;
use Kopling\Core\Storage\StorageRequest;

class Extension extends AbstractExtension implements RequestsStorageDriver
{
    // ...name(), description() as above

    /** @return array<StorageRequest> */
    public function storage(): array
    {
        return [new StorageRequest()];
    }
}

You only implement the contracts relevant to your extension — Kopling checks for each one independently, so adding a new contract in a future release never affects extensions that don't use it.

RequestsStorageDriver

Declares a storage driver your extension provides. Example above.

HasPermissions

Declares granular, admin-grantable permissions. Section 6.

HasPortals

Registers a new named UI surface — its own route and layout. Section 7.

ChangesUx

Places UI, like a nav link, into an existing slot on any Portal. Section 8.

CannotBeDisabled

A marker — no method at all. See below.

CannotBeDisabled: a marker, not a capability

Kopling doesn't yet have a way for an admin to disable an installed extension at all — that logic is still to come. This contract exists ahead of it anyway, so the guarantee is in place from day one rather than retrofitted once disabling exists: whatever eventually implements that toggle must refuse for any Extension implementing this marker.

use Kopling\Core\Extension\Contract\CannotBeDisabled;

class Extension extends AbstractExtension implements CannotBeDisabled
{
    // ...name(), description() as above -- nothing else to implement
}

This isn't only Core's own escape hatch. A hosting provider bundling their install with an extension they don't want a community admin able to turn off — a required moderation tool, something their terms of service depend on — implements the same marker on their own extension, no different from Core doing it on itself.

Section 6

Permissions

If your extension needs to gate an action behind something an admin can grant or take away, declare a permission — don't hardcode a check against "is this person an admin." Kopling has no such flag: every permission is its own named, granular capability, and "Admin"/"Moderator" are just conventional bundles of them an admin assigns, never anything the code treats specially.

use Kopling\Core\Authorization\Permission;
use Kopling\Core\Extension\Contract\HasPermissions;

class Extension extends AbstractExtension implements HasPermissions
{
    // ...name(), description() as above

    /** @return array<Permission> */
    public function permissions(): array
    {
        return [
            new Permission(
                id: 'manage-things',
                label: __('yourvendor-example::permissions.manage-things.label'),
                description: __('yourvendor-example::permissions.manage-things.description'),
            ),
        ];
    }
}

Write only the local part of the id — manage-things, not yourvendor-example::manage-things. Kopling prefixes it with your extension's own id before it's ever registered, joined with :: — the same separator, and the same reason, as your view and translation namespaces (Section 4) — so you never have to worry about another extension picking the same name.

label/description go through your extension's own lang/ (Section 4) with Laravel's __() helper, the same as any other user-facing string — never a hardcoded string in whatever language you happened to write the extension in. By the time permissions() is called, your lang/ is already registered, so the translation resolves correctly.

Recommended naming

Verb-noun, lowercase, hyphenated — manage-things, view-reports, delete-things — matching the shape of core's own permissions (core::manage-people), so an admin's permission list reads consistently across every installed extension instead of as a patchwork of each author's own style.

The callback is an escape hatch, not the default

Most permissions need nothing beyond a name — Kopling checks whether the signed-in person's groups have been granted it, and that's the whole decision. For the rare case that needs more (e.g. "can edit this specific reply," not any reply), Permission takes an optional callback that runs in addition to the grant check, never instead of it — it can only narrow access down further, never grant it on its own.

Section 7

Portals

A Portal is a named UI surface — a route prefix paired with the Blade layout its routes render inside. Kopling ships two: core::community (path '', the default site) and core::admin (path admin). Neither is special-cased — a Portal is a registrable pattern any extension can add its own instance of, the same way a Moderation portal for large communities is expected to arrive later as a first-party extension (the charter's own proof case for this system, Section 9 & D29).

Most extensions never need to register a Portal at all — putting a link into an existing Portal (Section 8, UI slots) is the common case. Reach for HasPortals only when you're building a genuinely separate surface — its own route, its own layout, its own optional gate on entry — not just another page inside the Community or Admin portal.

A Portal is not a gating mechanism

Registering a Portal never replaces checking permissions on your own routes. Kopling has no "is this person an admin" flag anywhere — every action still checks its own granular Permission, exactly as it would outside a Portal. A Portal's own optional gate (below) covers only that Portal's single root route; anything else your extension registers under that same path prefix gates itself independently, the normal way.

use Kopling\Core\Extension\Contract\HasPortals;
use Kopling\Core\Portal\Portal;

class Extension extends AbstractExtension implements HasPortals
{
    // ...name(), description() as above

    /** @return array<Portal> */
    public function portals(): array
    {
        return [
            new Portal(
                id: 'moderation',
                label: 'Moderation',
                path: 'moderation',
                layout: 'yourvendor-example::layouts.moderation',
                permission: 'moderate',
            ),
        ];
    }
}

As with permission ids (Section 6), write only the local part — moderation, not yourvendor-example::moderation. Kopling prefixes it the same way, for the same collision-safety reason: your Portal ends up registered and routable as yourvendor-example::moderation, at /moderation.

Portal fields

FieldWhat it's for
idLocal id, auto-prefixed with your extension's id — becomes the route name.
labelDisplay name, shown in the Portal's header and anywhere Kopling lists Portals.
pathThe route path your Portal is registered at, e.g. 'moderation' → /moderation. '' registers at the root.
layoutA Blade view name — conventionally a thin wrapper delegating to <x-k::portal.layout> (below).
permissionOptional. A local permission id, auto-prefixed, that gates this Portal's own root route only — leave it null to leave the root open.
middlewareOptional. Overrides the default web middleware group on the Portal's root route.
iconOptional. Reserved for representing the Portal visually; not yet wired up anywhere.
descriptionOptional. A one-line description of what the Portal is for.

What registering a Portal actually does

Kopling registers exactly one route per Portal: a GET at path, named after the fully-prefixed id. Visiting it checks permission (if you set one) against the signed-in person, then renders layout with the Portal instance bound to it as $portal — the same mechanism Kopling's own Community and Admin portals run through, no special case for either.

Your layout is expected to be a thin wrapper, the same way Kopling's own are:

<!-- yourvendor-example/views/layouts/moderation.blade.php -->

<x-k::portal.layout portal="yourvendor-example::moderation">
    @yield('content')
</x-k::portal.layout>

<x-k::portal.layout> is the one shared shell every Portal renders through — head, a side-navigation region, a header showing label, and a main content slot — so individual Portal layouts never duplicate that chrome. What actually shows up in the side navigation isn't decided by which Portal it is; every Portal renders the exact same core::side-navigation slot, filled by whatever ChangesUx entries pass their own permission check (Section 8) — that's true for the Admin portal today, and it's true for any Portal you register too.

Section 8

UI slots

Putting your own UI into a Portal you don't own — a link in the Admin portal's side navigation, for instance — is the ChangesUx contract, not HasPortals. This is the common case: most extensions place a few things into an existing Portal and never register one of their own.

use Kopling\Core\Extension\Contract\ChangesUx;
use Kopling\Core\Ux\Portal\Navigation\Item;
use Kopling\Core\Ux\Ux;

class Extension extends AbstractExtension implements ChangesUx
{
    // ...name(), description() as above

    public function ux(): Ux
    {
        return Ux::make()
            ->add(Item::class, ['label' => 'Hello', 'route' => 'example.hello'])
            ->in('core::side-navigation')
            ->as('hello')
            ->after('core::theme')
            ->when('manage-things');
    }
}

Read as a sentence: place an Item component, with this data, into the core::side-navigation slot, right after Core's own entry, visible only to people with manage-things.

Ux::make() chain

MethodWhat it's for
add($component, $data)Starts a new entry. $component is either an already-valid Blade tag ('k::portal.navigation.item') or the component's own class (Item::class, as above) — Kopling resolves a class back to its tag for you, so reach for whichever reads better. $data is the array it receives as its one data prop.
in($slot)The slot to place it in. Write the full, already-namespaced slot name yourself — 'core::side-navigation', not 'side-navigation'. Unlike every other id on this page, Kopling never prefixes this one: a slot is a public rendezvous point every extension needs to be able to name exactly, the same way, regardless of which extension is naming it.
as($id)Optional. Names this entry so other extensions can anchor before/after it, or replace/remove it. Defaults to the component string if omitted. Auto-prefixed with your extension's id, same as permissions.
after($id) / before($id)Optional. Anchors this entry's position relative to another entry's already-prefixed id (e.g. 'core::theme').
when($condition)Optional. A local permission id, auto-prefixed — the entry only renders for people who pass it. Leave it off and the entry always renders.
edit($id)Re-selects an entry you already added earlier in this same chain so you can configure it further — for when you're not done with an entry but have since moved on to add()ing others. Throws if nothing in this chain has that id; this is always a typo, never an install-order concern.
replace($id, $component, $data)Overwrites another entry's component/data by its already fully-qualified id — Core's own 'core::theme', or anything registered by an extension that loads before yours. Anything you don't also chain here (in()/when()/etc.) keeps the original entry's value; a missing target is a no-op, same as a dangling after().
remove($id)Removes another entry outright, by the same already fully-qualified id rule as replace(). Also a no-op if the target doesn't exist.

You can only replace()/remove() something registered by an extension that Kopling loads before yours (Core always loads first) — never one that loads after, since it hasn't registered anything yet when yours runs. This is the same ordering constraint Laravel's own service providers already have, not something new this system adds.

Exactly one slot exists today: core::side-navigation, rendered inside every Portal's shared layout (Section 7). More slots — head assets, post actions, admin widgets — are planned as real needs prove them out, not built yet (Section 11).

Section 9

Icon

An icon/ directory with a square PNG represents your extension anywhere Kopling lists installed extensions.

icon/lg.png

Required if you want an icon at all — this is the default, shown at 2048×2048.

icon/sm.png

Optional. Extensions can ship additional named sizes for smaller display contexts; only lg.png is required.

Section 10

The reference implementation

k-extensions/example in the kopling/development monorepo is a dummy extension — not meant to do anything real — kept specifically so every convention on this page has a working, verified example behind it: every path, the Extension class, the RequestsStorageDriver, HasPermissions, and ChangesUx contracts, and both icon sizes. Copy it as a starting point rather than assembling one from this page alone.

HasPortals (Section 7) and CannotBeDisabled (Section 5) aren't demonstrated by the reference implementation yet — only Core registers its own Portals and marks itself undisableable so far. Ux::replace()/remove() (Section 8) aren't either — there isn't yet a real second extension with a reason to override the first one's entries. If this page and that directory ever disagree, the code is right and this page is stale — please tell us.

Section 11

Still evolving

This is a living document. Nothing here is final — reference it, don't assume it's the last word.

How css/ and js/ actually reach the page — a core-owned outlet is planned, not yet built.
UI slots beyond core::side-navigation — head assets, post actions, admin widgets — as real needs prove them out; ChangesUx (Section 8) supports any named slot, but only one is wired up today.
No first-party extension registers its own Portal (Section 7) yet — the Moderation portal named in the charter (D29) is the planned proof case.
No way for an admin to disable an installed extension exists yet at all — CannotBeDisabled (Section 5) is the guard for that future toggle, built ahead of the toggle itself so the guarantee is never missing on day one.
More contract interfaces beyond RequestsStorageDriver, HasPermissions, HasPortals, ChangesUx, and CannotBeDisabled, as further real needs (listeners, settings) prove themselves out.
Kopling\Extend\* — a wrapped, blessed subset of Laravel primitives (Eloquent, validation, events) extensions will use directly, instead of touching Laravel itself. Not yet populated.
Multiple icon sizes beyond lg.png/sm.png — naming convention not yet fully settled.
No admin UI yet for assigning permissions to groups — only a PHP API (Group::givePermissionTo()/revokePermissionTo()).
